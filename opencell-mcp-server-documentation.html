1. Context & Motivation

Opencell is a Java EE/Jakarta EE billing platform with ~93 REST API v2 endpoints covering the full quote-to-cash lifecycle: quotes, orders, invoicing, payments, dunning, catalog management, mediation, and more.

The Challenge With ~93 resources × ~5 operations each = ~465 potential tools. Exposing all of them individually to an LLM would overwhelm the context window and degrade tool selection accuracy.

The Solution: A three-tier MCP server that reduces 465 potential tools down to ~30 well-organized tools:

  • Tier 1: Generic CRUD tools leveraging Opencell's existing /v2/generic endpoint
  • Tier 2: Domain-specific action tools for non-CRUD business operations
  • Tier 3: Discovery & utility tools for entity introspection

Consumers:

  • AI agents (Claude Code, Claude Desktop, etc.) via stdio transport
  • Internal tooling and automation pipelines via HTTP/SSE transport

Tech Stack: TypeScript, @modelcontextprotocol/sdk, Zod, Node.js 22+

Three-Tier Tool Model

Tier Purpose Count Strategy
Tier 3 Discovery & Utility 4 Entity introspection, version info, help guide
Tier 1 Generic CRUD 5 Wraps /v2/generic endpoint — works with any entity
Tier 2 Domain Actions 12 Business operations that can't be expressed as generic CRUD
Key Insight Opencell already has a generic CRUD API at /v2/generic that handles any entity by name. By wrapping it, we avoid duplicating 400+ individual CRUD tools. An LLM discovers entities at runtime via list_entities and describe_entity, then operates on them with generic tools.

Tool NameDescriptionKey Parameters

generic_delete Delete an entity by name and ID entityName, id
Generic API Contract The generic tools call the Opencell /v2/generic API:
  • POST /generic/all/{entityName} – list with GenericPagingAndFiltering body
  • POST /generic/{entityName}/{id} – get by ID
  • POST /generic/{entityName} – create (JSON body)
  • PUT /generic/{entityName}/{id} – update (JSON body)
  • DELETE /generic/{entityName}/{id} – delete
Entity names are case-insensitive. The GenericPagingAndFiltering DTO supports: genericFields, nestedEntities, nestedDepth, filters, fullTextFilter, offset, limit, sortBy, sortOrder, excluding, groupBy, having, and more.

5. Implementation Details

5.1 Configuration

All configuration is via environment variables, validated at startup with fail-fast behavior.

# .env.example

# Opencell API
OPENCELL_BASE_URL=https://opencell.example.com/opencell/api/rest
OPENCELL_API_VERSION=v2

# Keycloak Authentication
KEYCLOAK_URL=https://keycloak.example.com
KEYCLOAK_REALM=opencell
KEYCLOAK_CLIENT_ID=opencell-mcp
KEYCLOAK_CLIENT_SECRET=changeme

# Transport
MCP_TRANSPORT=stdio          # "stdio" or "http"
MCP_HTTP_PORT=3000

# Behavior
MCP_DEFAULT_PAGE_SIZE=50
MCP_MAX_PAGE_SIZE=500
MCP_REQUEST_TIMEOUT_MS=30000
MCP_LOG_LEVEL=info

The Config interface provides typed access to all settings:

interface Config {
  opencellBaseUrl: string;       // Base URL of Opencell REST API
  opencellApiVersion: string;    // API version (default: "v2")
  keycloakUrl: string;           // Keycloak server URL
  keycloakRealm: string;         // Keycloak realm
  keycloakClientId: string;      // OAuth client ID
  keycloakClientSecret: string;  // OAuth client secret
  mcpTransport: "stdio" | "http";
  mcpHttpPort: number;
  defaultPageSize: number;
  maxPageSize: number;
  requestTimeoutMs: number;
  logLevel: "debug" | "info" | "warn" | "error";
}

5.2 Authentication (OAuth/Keycloak)

The KeycloakClient manages the full OAuth token lifecycle:

First Request | v +-----------+ +------------------+ +-----------+ | Has valid | YES | Return cached | | Opencell | | token? |---->| access_token |---->| API Call | +-----------+ +------------------+ +-----------+ | NO v +-----------+ +------------------+ | Has valid | YES | POST /token | | refresh? |---->| grant=refresh |---+ +-----------+ +------------------+ | | NO | v v +------------------+ +------------------+ | POST /token | | Cache new token | | grant=client_cred|----------->| (30s early | +------------------+ | expiry buffer) | +------------------+

Key behaviors:

  • Token URL: {KEYCLOAK_URL}/realms/{REALM}/protocol/openid-connect/token
  • Grant type: client_credentials
  • Tokens cached in memory with 30-second early expiry buffer
  • Automatic refresh using refresh token when available
  • Falls back to full re-authentication if refresh fails

5.3 HTTP Client

The OpencellClient wraps fetch() with:

  • Automatic Authorization: Bearer {token} header injection
  • Request timeout via AbortController (configurable, default 30s)
  • Automatic retry on 401 (re-authenticates and retries once)
  • JSON serialization/deserialization
  • URL construction with query parameter support
// Methods available
client.get(path, queryParams?)
client.post(path, body?, queryParams?)
client.put(path, body?, queryParams?)
client.delete(path, queryParams?)

// Returns
interface HttpResponse {
  status: number;
  data: unknown;
  headers: Record<string, string>;
}

Paginated list responses are formatted for LLM readability:

Found 142 total results (showing 1 to 50):

Use offset=50 to see more results.

5.5 Transport

stdio (default)

Uses StdioServerTransport from the MCP SDK. Ideal for Claude Code and CLI-based AI agents.

HTTP/SSE

Uses SSEServerTransport with a Node.js HTTP server. Supports multiple concurrent sessions.

  • GET /sse – Establish SSE connection (returns session ID)
  • POST /messages?sessionId={id} – Send messages to a session
  • GET /health – Health check endpoint

8. Usage Guide

8.1 Build

cd opencell-mcp-server
npm install
npm run build

8.2 Run (stdio)

# Set environment variables
export OPENCELL_BASE_URL=https://your-instance/opencell/api/rest
export KEYCLOAK_URL=https://your-keycloak
export KEYCLOAK_REALM=opencell
export KEYCLOAK_CLIENT_ID=opencell-mcp
export KEYCLOAK_CLIENT_SECRET=your-secret

npm start

8.3 Run (HTTP/SSE)

export MCP_TRANSPORT=http
export MCP_HTTP_PORT=3000
npm start
# Server listening on http://localhost:3000
# SSE endpoint: http://localhost:3000/sse
# Health check: http://localhost:3000/health

8.5 Typical AI Agent Workflow

  1. Discover: list_entities → shows all available entities (Invoice, Customer, BillingAccount, etc.)
  2. Inspect: describe_entity({entityName: "Invoice"}) → shows fields, types, relationships
  3. Query: generic_list({entityName: "Invoice", filters: {status: "VALIDATED"}, limit: 10})
  4. Read: generic_get({entityName: "Invoice", id: 123})
  5. Act: invoice_actions({action: "cancel", invoiceId: 123})

11.2 Testing Checklist

Phase 1 Tests

  • Configure Claude Code to use the MCP server via stdio
  • Run list_entities — verify entity names returned
  • Run describe_entity for Invoice — verify fields and types
  • Run generic_list for invoices — verify paginated results
  • Run generic_get for a specific invoice — verify full entity data

Phase 2 Tests

  • Ask Claude to "cancel invoice #123" — verify it uses invoice_actions with action=cancel
  • Ask Claude to "transfer subscription X to billing account Y" — verify account_actions
  • Ask Claude to "create a payment by card" — verify payment_actions

HTTP Transport Tests

  • Start with MCP_TRANSPORT=http
  • Connect via SSE from a test client
  • Verify tool listing and execution
  • Test concurrent sessions

Auth Tests

  • Verify token acquisition on first request
  • Set short token TTL, verify automatic refresh
  • Verify retry on 401 during request
